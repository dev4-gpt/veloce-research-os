## Decision Summary

The designs are useful, but v1.2 implementation should be narrower and more conservative than the Paperclip drafts.

Implementation order:

```text
1. Build a tiny status_check OpenAPI tool server.
2. Register that tool server in Open WebUI.
3. Keep Hermes exposed to Open WebUI through the existing OpenAI-compatible chat endpoint.
4. Do not implement Paperclip-to-Hermes inside Paperclip yet.
5. Document a Paperclip-to-Hermes HTTP design that uses the verified endpoint.
```

## Review of v1.2A: Paperclip to Hermes HTTP Adapter

Useful parts:

- Correctly identifies that the local `hermes` command path is broken.
- Correctly proposes HTTP auth through a bearer token.
- Correctly requires structured issue comments and failure mapping.
- Correctly says implementation should not happen from Paperclip's restricted workspace.

Required correction:

The design proposes:

```text
POST /v1/tasks/run
```

That endpoint is not the endpoint we verified.

The verified Hermes endpoint is:

```text
POST https://hermes.srv1314350.hstgr.cloud/v1/chat/completions
```

Minimum working request:

```json
{
  "model": "hermes-agent",
  "messages": [
    {
      "role": "user",
      "content": "Task prompt goes here."
    }
  ]
}
```

Decision:

```text
Do not build a hypothetical /v1/tasks/run adapter in v1.2.
If Paperclip-to-Hermes is implemented, it must target /v1/chat/completions first.
```

## Review of v1.2B: Open WebUI MCPO Tool Layer

Useful parts:

- Correctly starts with a safe `status_check` tool.
- Correctly avoids broad shell execution.
- Correctly recommends strict target allowlists.
- Correctly keeps the tool internal to the Docker network where possible.

Required correction:

The first v1.2 tool does not need full MCP/Ruflo.

Open WebUI can consume OpenAPI-compatible tool servers. MCPO is useful when bridging an MCP server into OpenAPI, but a simple `status_check` tool can be implemented directly as an OpenAPI HTTP server.

Decision:

```text
Build direct OpenAPI status_check server first.
Add MCPO only after one direct OpenAPI tool works in Open WebUI.
```

This avoids repeating the prior Ruflo image failure.

## Review of v1.2C: Operator Cockpit Workflow

Useful parts:

- Clear distinction between Open WebUI, Paperclip, Hermes, direct NVIDIA, GitHub, VPS, and Obsidian.
- Good default tool sequence:
  - load task
  - run task
  - persist evidence
- Good operator checklist.
- Good pinned Open WebUI operator prompt.

Decision:

```text
Accept v1.2C as the operating model.
Use it to configure the Open WebUI operator prompt after status_check works.
```

## V1.2 Implementation Plan

### Phase 1: Add Status Check OpenAPI Tool Server

Add a small service to the infrastructure stack in `/root/ai-agency`.

Purpose:

```text
Let Open WebUI call one safe tool that reports stack status.
```

Tool name:

```text
status_check
```

Allowed targets:

```text
openwebui
hermes
paperclip
research_repo
```

Response shape:

```json
{
  "ok": true,
  "target": "hermes",
  "checked_at": "2026-05-18T00:00:00Z",
  "detail": "Hermes health endpoint returned ok",
  "trace_id": "..."
}
```

Security rules:

- No arbitrary shell command execution.
- No arbitrary URL fetching.
- No secrets in responses.
- Fixed allowlist only.
- Internal Docker network first.

### Phase 2: Register Tool in Open WebUI

Open WebUI admin action:

```text
Register OpenAPI tool server URL.
```

Preferred internal URL:

```text
http://status-tool:8080/openapi.json
```

If Open WebUI cannot reach the internal URL, expose through Traefik as:

```text
https://tools.srv1314350.hstgr.cloud/openapi.json
```

Use authentication if exposed publicly.

### Phase 3: Add Operator Prompt

Use the v1.2C pinned prompt in Open WebUI after the tool is visible.

Initial operator instruction:

```text
Use direct NVIDIA models for fast drafting and analysis.
Use hermes-agent when memory or agent behavior matters.
Use status_check before assuming the stack is healthy.
Use Paperclip for task tracking, not code implementation.
Use GitHub and VPS as the source of truth for code.
```

### Phase 4: Revisit Paperclip-to-Hermes

After Open WebUI tool calling works, decide whether Paperclip-to-Hermes is still needed.

If needed, implement a small external adapter service instead of modifying Paperclip internals first.

Safer adapter shape:

```text
Paperclip issue/comment -> webhook/adapter service -> Hermes /v1/chat/completions -> Paperclip comment
```

This keeps Hermes integration outside Paperclip source code and avoids the local `hermes` command path.

## V1.2 Acceptance Criteria

V1.2 is complete when:

```text
Open WebUI can call status_check.
status_check can report Open WebUI/Hermes/Paperclip/research_repo status.
Hermes remains verified for standalone chat and memory.
The Open WebUI operator prompt is installed or documented.
Paperclip-to-Hermes is either implemented through the verified endpoint or explicitly deferred to v1.3.
SYSTEM_STATUS.md is updated.
```

## What Not To Do

Do not:

- Build broad shell execution into Open WebUI.
- Expose internal services without auth.
- Use `/v1/tasks/run` unless Hermes actually exposes it later.
- Retry the broken Paperclip local Hermes adapter.
- Let Paperclip agents implement code directly inside `/root/veloce-research-os`.
- Reintroduce Ruflo until the basic OpenAPI tool path is working.
