# V1.4 Hermes, MCPO, and Ruflo Integration Plan

Date: 2026-05-18

## Goal

Make the deployed Veloce stack usable without repeating the Paperclip recovery-loop pattern.

V1.4 should improve flow, not add more agents for their own sake.

## Decisions

1. Hermes remains the memory/agent-behavior lane.
2. Direct NVIDIA/Open WebUI models remain the fast drafting and exact-answer lane.
3. Paperclip remains the task board and artifact ledger.
4. MCPO is added after the direct Open WebUI status tool because Open WebUI documentation recommends MCPO for bridging stdio/SSE MCP servers into OpenAPI-compatible tool servers.
5. Ruflo is a gated experiment, not a default runtime dependency.

## Why Hermes Looped

The Hermes HTTP bridge is working at the connectivity level:

```text
Paperclip -> /paperclip/adapters/hermes_http_agent_env.sh -> Hermes /v1/chat/completions
```

The repeated failures came from issue disposition and prompt-shape mismatch:

- Paperclip wanted exact output.
- Hermes received large runtime/recovery context.
- Hermes returned successful but non-exact answers.
- Paperclip created recovery issues because the issue status was not changed to a valid disposition quickly enough.

## V1.4A: Reduce Hermes/Paperclip Token Bloat

Implemented repo support:

```text
tools/paperclip/hermes_http_agent.mjs
tools/paperclip/hermes_http_agent_env.sh
tools/paperclip/hermes.env.example
```

New controls:

```text
HERMES_MAX_PROMPT_CHARS=12000
HERMES_SYSTEM_PROMPT=brief agent instruction
HERMES_FIXED_PROMPT=optional deterministic test prompt
```

Operating rule:

```text
Do not use Hermes for exact tiny replies.
Use Hermes for memory, agent behavior, and synthesis where its added context helps.
Use direct models for exact confirmations and cheap fast work.
```

## V1.4B: Add MCPO Safely

MCPO should start with one boring MCP server:

```text
mcp-server-time
```

Why:

- It proves MCP -> MCPO -> OpenAPI -> Open WebUI without giving shell or filesystem access.
- It has low security risk.
- It is easy to recognize when it works.

Verified VPS state as of 2026-05-19:

```text
Container: aiagency-mcpo-time
Status: healthy
Internal OpenAPI URL: http://mcpo-time:8000/openapi.json
Paths: /get_current_time, /convert_time
Direct call: /get_current_time with America/New_York returned valid time JSON
```

Repo support:

```text
tools/mcpo/Dockerfile
deploy/ai-agency/docker-compose.mcpo.yml
docs/mcpo-ruflo-setup.md
```

Done criteria:

```text
aiagency-mcpo-time is healthy.
Open WebUI can import the MCPO OpenAPI endpoint.
Open WebUI can call the time tool through a tool-capable model.
SYSTEM_STATUS.md records the result.
```

## V1.4C: Ruflo Gate

Ruflo should not be enabled as part of the default daily flow until MCPO is proven.

Reasons:

- Ruflo is an orchestration layer with many tools and agent behaviors.
- The current problem is already token bloat and runaway recovery loops.
- Adding another broad orchestration runtime before MCPO is stable would make debugging harder.

Gate criteria before Ruflo is allowed:

```text
1. MCPO tool call works in Open WebUI chat, not only direct curl.
2. Ruflo image/build path is confirmed.
3. Ruflo is isolated from production Paperclip tasks.
4. One low-risk Ruflo capability is selected.
5. A rollback command is documented.
```

## Human Run Order

Run these phases one at a time:

```text
1. Pull repo on VPS.
2. Install updated Hermes wrapper.
3. Verify Paperclip Hermes wrapper once.
4. Stop all Hermes exact-output retry issues.
5. Bring up MCPO time bridge.
6. Register MCPO in Open WebUI.
7. Verify one MCPO tool call from Open WebUI chat.
8. Decide whether Ruflo passes the gate.
```

## Anti-Loop Rule

If Paperclip creates a recovery issue for a successful run:

```text
Mark the source issue Done, In Review, Blocked, or Cancelled manually.
Then mark the recovery issue Done.
Do not wake the same agent more than once.
```
