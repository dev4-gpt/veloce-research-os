# V1.9N Manifest Penalty And Markdown Priority

Date: 2026-05-20

## Context

V1.9M correctly promoted `knowledge/graph-memory/` into graph query results, but the first live query showed an overcorrection: `knowledge/graph-memory/manifest.json` fields such as `mcpo_paths`, `openwebui_tools`, and `paperclip_items` ranked above the human-readable memory notes.

That proved the graph-memory boost worked, but the product needs readable operating-memory pages to lead.

## What Changed

V1.9N changes the ranking behavior:

- `source_filter="docs"` excludes `knowledge/graph-memory/manifest.json`.
- `source_filter="knowledge"` keeps manifest data available, but penalizes it.
- Markdown files under `knowledge/graph-memory/*.md` receive an additional boost.
- `knowledge/graph-memory/veloce-operating-graph.md` receives a targeted boost for operating-graph questions.
- The ranking mode is now reported as `v1.9N_weighted_<source_filter>_<intent>`.

## Preferred Top Evidence

For operating graph questions, the expected top results are human-readable memory files:

```text
knowledge/graph-memory/veloce-operating-graph.md
knowledge/graph-memory/openwebui-mcpo-tool-surface.md
knowledge/graph-memory/paperclip-work-ledger.md
knowledge/graph-memory/hermes-ruflo-skills.md
knowledge/graph-memory/obsidian-graphify-memory-loop.md
docs/v1.9L-operating-graph-ingestion.md
```

## Why This Matters

The manifest is useful for automation, but it is not the product story. OpenWebUI, Hermes, Ruflo, and a human in Obsidian should see the readable operating memory first, then inspect manifest internals only when needed.

This makes the graph behave more like an operating memory layer and less like a raw extraction dump.
