# V1.8 Unattended Autonomous Production Plan

Date: 2026-05-20

## Current Answer

Veloce Research OS is ready for controlled operator workflows.

It is not ready for unattended autonomous production until the stack has a closed-loop autonomy controller with scoped permissions, durable audit logs, rollback automation, alerting, and kill switches.

Current safe status:

```text
Operator beta: yes
Human-approved production changes: yes
Unattended autonomous production: blocked
Customer/SLA autonomous operation: blocked
```

## Definition

Unattended autonomous production means an agent can, without a human present in the loop:

```text
1. Detect work or incidents.
2. Decide whether action is allowed.
3. Change production-facing state.
4. Verify the change.
5. Roll back if verification fails.
6. Record evidence in Paperclip/GitHub/Obsidian.
7. Alert the operator when risk or failure thresholds are crossed.
```

This is a different safety class from the current system, where Open WebUI and MCPO provide visibility and planning while Codex/GitHub/VPS remain human-approved execution paths.

## Autonomy Architecture

Target production shape:

```text
Scheduler / Event Source
  -> Autonomy Controller
    -> Policy Gate
      -> Capability Broker
        -> allowlisted tools only
      -> Execution Sandbox
      -> Verification Runner
      -> Rollback Runner
      -> Audit Ledger
      -> Alert Sink
```

Core rule:

```text
No model receives raw production credentials.
No model gets arbitrary shell, Docker, filesystem, GitHub, or Paperclip mutation access.
Every action is mediated by a typed capability with policy, budget, evidence, and rollback metadata.
```

## Required Controls

### Secret Rotation and Isolation

Required before any autonomous write path:

```text
MCPO_API_KEY rotated after exposed compose output.
OPENWEBUI_SECRET_KEY rotated.
OPENWEBUI_ADMIN_PASSWORD rotated.
API_SERVER_KEY rotated.
NVIDIA_API_KEY rotated.
PAPERCLIP_API_KEY provisioned as scoped write token, not broad operator token.
Alert webhook secret provisioned only after redaction checks exist.
```

Secrets must be injected only into trusted services, never echoed into agent prompts, reports, compose render output, or Paperclip comments.

### Capability Broker

Replace direct tool access with typed capability calls:

```text
read_stack_status
read_repo_status
read_ruflo_status
create_execution_packet
create_github_branch
open_pull_request
run_verification_suite
deploy_canary
promote_canary
rollback_canary
post_paperclip_comment
set_paperclip_disposition
send_alert
```

Forbidden in unattended mode:

```text
arbitrary shell
raw Docker control
raw filesystem writes
raw GitHub token access
raw Paperclip database/API access
secret reads
Ruflo daemon/swarm/hooks/autopilot execution
browser extension production writes
```

### Policy Gate

Every proposed action must be classified:

```text
read_only: allowed unattended
comment_only: allowed unattended after Paperclip scoped-token review
low_risk_write: allowed only through canary path
production_write: requires rollout policy, verification, rollback, and alerting
destructive: human approval required
secret_bearing: human approval required
```

### Canary and Rollback

Unattended production writes require:

```text
separate branch or isolated config candidate
canary deploy or dry-run render
pre-change health snapshot
post-change health snapshot
automatic rollback command
rollback verification command
operator alert on rollback or verification failure
```

### Audit Ledger

Every autonomous action writes a structured ledger record:

```text
action_id
issue_id
actor
requested_capability
policy_decision
inputs_hash
outputs_hash
commit_sha
verification_command
verification_result
rollback_command
rollback_result
alert_result
disposition
```

The ledger must avoid secrets and must be durable outside the running container.

### Monitoring and Alerting

Autonomy requires active alerting, not only manual `make stack-health`:

```text
scheduled stack-health
failure alert destination
rollback alert destination
stale issue alert
failed disposition alert
budget/loop alert
provider spend alert
```

### Paperclip Control Plane

Paperclip cannot be an unattended production controller until issue mutation is reliable and scoped.

Required:

```text
POST issue comments works from the authorized automation identity.
PATCH issue status works from the authorized automation identity.
Recovery issues are deduplicated or suppressed after parent is done.
Agent runs always end with one valid disposition.
Paperclip API writes are rate-limited and audited.
```

### Ruflo Production Boundary

Ruflo remains planning/packet only in v1.8.

Ruflo production execution can be reconsidered only after the capability broker, policy gate, audit ledger, and rollback runner are already passing without Ruflo runtime execution.

## V1.8 Task Sequence

### V1.8A - Autonomy Readiness Gate

Goal: add a reproducible gate that reports whether unattended production is blocked or ready.

Deliver:

```text
autonomy_readiness_v1_8.json
autonomy_readiness_gate.py
make autonomy-readiness
generated JSON/Markdown report
```

Acceptance:

```text
Gate fails closed while blockers exist.
Report lists every blocker with owner, action, and evidence.
No secrets printed.
```

### V1.8B - Secret Rotation and Redaction Validation

Goal: rotate exposed secrets and prove redaction paths.

Acceptance:

```text
All exposed secrets rotated.
Containers redeployed.
make stack-health passes.
make rollback-drill output remains suppressed/redacted.
No generated artifact contains secret-looking values.
```

### V1.8C - Scoped Paperclip Automation Token

Goal: make Paperclip comment/status mutation reliable without broad access.

Acceptance:

```text
Scoped token can post comments and set dispositions.
Token cannot read or print secrets.
Recovery issue loop is controlled.
All writes produce audit records.
```

### V1.8D - Capability Broker MVP

Goal: expose only typed, policy-checked automation capabilities.

Acceptance:

```text
No arbitrary shell capability.
No raw Docker capability.
No raw GitHub token capability.
All capabilities have input schema, output schema, risk level, rollback metadata, and evidence path.
```

### V1.8E - Autonomy Controller Dry Run

Goal: run the controller without production writes.

Acceptance:

```text
Controller reads one Paperclip issue.
Creates a plan.
Creates an execution packet.
Runs stack-health.
Posts a comment only if scoped Paperclip write is approved.
Does not change GitHub, Docker, VPS, or Ruflo runtime state.
```

### V1.8F - Canary Write Path

Goal: allow one low-risk unattended write through branch/PR only.

Acceptance:

```text
Agent creates branch.
Agent makes non-secret doc/config change.
Agent runs tests.
Agent opens PR.
Human merge remains required.
```

### V1.8G - Limited Production Promotion

Goal: allow an unattended low-risk deploy only after canary and rollback controls pass.

Acceptance:

```text
Only pre-approved deploy target.
Pre-change snapshot captured.
Canary deploy succeeds.
Stack health succeeds.
Rollback drill succeeds.
Audit ledger record complete.
Operator receives success/failure alert.
```

## Production Gate

Unattended autonomous production is allowed only when:

```text
1. v1.8A autonomy-readiness gate passes.
2. v1.8B secret rotation is complete and verified.
3. v1.8C Paperclip scoped write auth is reliable.
4. v1.8D capability broker blocks raw shell/Docker/filesystem/GitHub writes.
5. v1.8E dry run succeeds without production mutation.
6. v1.8F canary branch/PR path succeeds.
7. v1.8G limited production promotion succeeds with rollback proof.
8. Active alerting is configured.
9. Audit ledger is durable and secret-free.
10. A human can disable autonomy with one documented command.
```

## Immediate Recommendation

Start with v1.8A and v1.8B.

Do not enable unattended production writes until the readiness gate reports pass and the exposed secrets from the earlier compose-output run have been rotated.
