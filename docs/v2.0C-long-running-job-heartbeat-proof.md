# V2.0C Long-Running Job Heartbeat Proof

V2.0C proves the job-control layer that should exist before canary deploy or rollback automation.

The goal is intentionally narrow:

```text
approved job intent -> resumable job packet -> heartbeat ledger -> stale-job detector -> audit ledger -> graph memory packet
```

This is not autonomous production execution. It is the minimum durable job loop required for responsible unattended work.

## Default Dry Run

```bash
cd projects/reliability-policy-matrix
make long-running-job-proof
```

The default config is:

```text
configs/long_running_job_v2_0C.json
```

By default, `live_enabled` is `false`. The proof writes only local derived artifacts.

## Live Gates

The proof has no external token requirement, but live-ready mode is still blocked unless both gates pass:

```text
VELOCE_LONG_JOB_LIVE=1
live_enabled=true
```

This keeps the same promotion discipline as V2.0A and V2.0B.

## Scoped Actions

The proof writes:

```text
one job packet
one heartbeat record
one stale-job detector pass
one audit ledger record
one Obsidian/Graphify memory packet
```

The stale detector includes a synthetic stale probe so the proof demonstrates that stale jobs are noticed before canary or rollback automation is attempted.

## Outputs

```text
artifacts/derived/long_running_job_v2_0C.json
artifacts/derived/long_running_job_v2_0C.md
artifacts/derived/long_running_job_audit_v2_0C.jsonl
artifacts/derived/long_running_job_heartbeat_v2_0C.jsonl
artifacts/derived/long_running_job_memory_v2_0C.md
artifacts/derived/v2C_jobs/
```

The memory markdown is intended for the Obsidian -> Graphify loop, so the proof becomes queryable through OpenWebUI graph memory after extraction.

## Rollback

If a job should not continue, the rollback is:

```text
cancel the queued job
release the lease
mark the audit disposition blocked
rerun graph extraction so memory reflects the correction
```

After V2.0C is accepted, Veloce can move to chat-to-canary with a real heartbeat and stale-job control already in place.
