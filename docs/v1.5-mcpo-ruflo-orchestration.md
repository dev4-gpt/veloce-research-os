# V1.5 MCPO and Ruflo Orchestration

Date: 2026-05-19

## Purpose

V1.5 turns the working Veloce stack into a more usable operator flow:

```text
Open WebUI -> safe tools through MCPO
Paperclip -> task board and durable issue ledger
Hermes -> memory and agent behavior
GitHub/VPS -> source of truth and verification
Ruflo -> gated orchestration experiment
```

The goal is not to add more moving parts for their own sake. The goal is to make the system coordinate work without repeating blocked Paperclip loops.

## Why MCPO

MCPO matters because Open WebUI needs a safe way to call tools, and most useful agent tools now arrive as MCP servers.

Without MCPO:

```text
Open WebUI can chat, but it cannot reliably call stdio/SSE MCP servers.
```

With MCPO:

```text
MCP server -> MCPO -> OpenAPI HTTP endpoint -> Open WebUI tool call
```

That gives us a narrow, inspectable bridge between chat and tools. It is safer than exposing shell access because every endpoint is explicit and allowlisted.

Current verified MCPO path:

```text
Open WebUI chat
-> Veloce MCPO Time native tool
-> http://mcpo-openwebui-proxy:8080/get_current_time
-> aiagency-mcpo-time
-> mcp-server-time
```

Verified proof:

```text
Open WebUI returned live JSON with datetime, trace_id, and latency_ms.
VPS logs showed POST /get_current_time HTTP/1.1 200.
```

## What To Add Through MCPO Next

Add tools in this order:

1. `time`
2. `stack_status`
3. `repo_status`
4. `artifact_index`
5. `paperclip_issue_lookup`

Do not add broad tools yet:

```text
No arbitrary shell.
No unrestricted filesystem.
No Docker control from chat.
No GitHub write access from chat.
No Paperclip mutation from chat.
```

The first expansion after time should be a read-only `stack_status` bridge that reports:

```text
Open WebUI health
Hermes health
Paperclip HTTP reachability
MCPO health
current git commit
recommended verification commands
```

## Why Ruflo

Ruflo is useful for a different layer than MCPO.

MCPO is the tool bridge.

Ruflo is the orchestration layer.

Use Ruflo when we need:

```text
multi-agent task routing
persistent orchestration memory
swarm-style coordination
workflow planning across agents/tools
conversational AI systems that call multiple capabilities
```

Ruflo should not replace Paperclip, Open WebUI, or Hermes. It should sit beside them as an orchestration experiment.

Recommended role:

```text
Ruflo coordinates plans.
MCPO exposes safe tools.
Open WebUI is the operator cockpit.
Paperclip records tasks and dispositions.
GitHub/VPS prove what actually changed.
```

## Ruflo Risk

Ruflo has a large surface area:

```text
agents
hooks
MCP tools
memory
workflow automation
possible shell and GitHub operations
```

That power is exactly why it should start isolated. The Veloce stack already had problems from agents retrying and creating recovery loops. Ruflo should be tested in a sandbox before it can coordinate production tasks.

## Ruflo Sandbox Gate

Ruflo may be evaluated only after these are true:

```text
1. MCPO time tool is verified from Open WebUI chat.
2. Ruflo install/start path is confirmed from the current upstream repo.
3. Ruflo has no production secrets by default.
4. Ruflo cannot write to /root/ai-agency or /root/veloce-research-os unless explicitly mounted read/write.
5. Only one low-risk orchestration capability is tested.
6. Rollback command is documented and tested.
```

Initial Ruflo test:

```text
Given a natural-language goal, produce a plan with owner, next action, verification command, and rollback note.
```

Do not let Ruflo execute the plan in the first test.

## Operator Flow

Daily default flow:

```text
1. Use Open WebUI with GPT OSS for tool-calling operations.
2. Enable Veloce MCPO tools in the chat.
3. Ask stack_status before assuming services are healthy.
4. Use Hermes only when memory or agent behavior matters.
5. Use Paperclip for task state and durable disposition.
6. Use GitHub/VPS for implementation and verification.
7. Use Obsidian for human-readable project memory.
```

## Decision

V1.5 should proceed with MCPO read-only tool expansion first.

Ruflo should be evaluated next, but only in planning-only sandbox mode.
