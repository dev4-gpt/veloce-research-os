# V1.9J Knowledge Graph Memory Loop

Date: 2026-05-20

## Purpose

Veloce should not treat the Graphify graph as a terminal-only artifact.

The product layer should let users ask questions about the project graph from OpenWebUI and should let important interactions become future graph memory through Obsidian and Graphify.

## Product Loop

```text
OpenWebUI interaction
-> Veloce Agentic Control
-> MCPO endpoint
-> secret-free memory record
-> Obsidian note / memory inbox
-> Graphify extraction
-> graphify-out/graph.json
-> OpenWebUI knowledge_graph_query
-> Hermes/Ruflo structured context
```

## Implemented Endpoints

### `/knowledge_graph_status`

Reads Graphify output from:

```text
graphify-out/graph.json
```

Returns:

```text
node count
edge count
community count
source file count
built_at_commit
GRAPH_REPORT.md path
```

### `/knowledge_graph_query`

Queries the local Graphify graph using keyword and relationship matching.

Example:

```json
{
  "question": "what connects OpenWebUI, Hermes, Ruflo, Paperclip, MCPO, Obsidian, and Graphify?",
  "max_results": 8
}
```

Returns:

```text
matching graph nodes
nearby relationships
evidence docs
next action for Hermes/Ruflo context
```

### `/knowledge_memory_record`

Creates a secret-free memory event packet that can be mirrored into Obsidian and re-indexed by Graphify.

Example:

```json
{
  "source_system": "openwebui",
  "event_type": "tool_call",
  "summary": "User queried Veloce Graphify status from chat.",
  "evidence_refs": ["docs/v1.9I-graphify-vps-proof.md"],
  "tags": ["graphify", "openwebui"],
  "dry_run": true
}
```

The endpoint rejects secret-like content before creating memory packets.

## Write Safety

By default, `/knowledge_memory_record` does not write files.

To enable writes in a mounted runtime volume:

```text
KNOWLEDGE_MEMORY_WRITE_ENABLED=true
KNOWLEDGE_MEMORY_DIR=/workspace/veloce-memory
```

This must only point to a dedicated memory volume, not the production repo checkout.

## OpenWebUI Tools

The OpenWebUI tool surface now includes:

```text
veloce_knowledge_graph_status
veloce_knowledge_graph_query
veloce_knowledge_memory_record
```

## Data Sources To Ingest

The graph memory layer should ingest only secret-free summaries from:

```text
OpenWebUI tool calls
Hermes memory/agent task summaries
Ruflo orchestration dry-run packets
MCPO status/query results
Paperclip issue closeouts
GitHub commits/PRs
VPS verification summaries
Obsidian operator notes
```

Do not ingest:

```text
.env files
raw secrets
raw provider tokens
browser profiles
private cookies/session data
raw docker compose config output
unredacted VPS logs
```

## Acceptance

```text
- OpenAPI exposes knowledge_graph_status.
- OpenAPI exposes knowledge_graph_query.
- OpenAPI exposes knowledge_memory_record.
- OpenWebUI tool exposes all three functions.
- Unit tests verify graph status/query behavior.
- Unit tests verify secret-like memory content is blocked.
- Unit tests verify Obsidian markdown packet generation.
```

## Product Claim

Veloce now has the beginning of an infinite-memory loop:

```text
human-readable memory in Obsidian
machine-readable memory in Graphify
chat access through OpenWebUI
agent reasoning through Hermes
planning context through Ruflo
audit and evidence through Paperclip/GitHub/VPS
```

This is not just a graph visualization. It is the foundation for a persistent operating memory layer.
