# V1.9 Full-Capability Agentic Product Plan

Date: 2026-05-20

## Thesis

Veloce should not stop at a safe operator workflow.

The product becomes compelling when a user can talk to one cockpit and the system can reliably coordinate:

```text
OpenWebUI chat
-> Hermes memory and agent reasoning
-> Ruflo planning and orchestration
-> capability broker
-> Paperclip task ledger
-> GitHub branch/PR path
-> VPS verification and deployment proof
-> rollback, alerting, audit ledger
```

The goal is not to remove humans from every decision. The goal is to make the system capable of doing the whole loop, while humans configure the policy boundary.

## Product Promise

```text
Describe work in chat. Veloce plans it, routes it, executes the allowed parts, verifies the result, records evidence, and asks for approval only when policy requires it.
```

## What Must Change

### 1. OpenWebUI Becomes The Product Surface

OpenWebUI must expose the same Veloce capabilities consistently:

```text
stack_status
repo_status
ruflo_status
ruflo_plan
ruflo_execution_packet
hermes_memory_query
hermes_agent_task
paperclip_issue_create
paperclip_comment
paperclip_disposition
github_pr_prepare
vps_verify
rollback_drill
autonomy_controls
autonomy_readiness
```

The user should not need to SSH for normal workflows.

### 2. Hermes Becomes A First-Class Reasoning/Memory Tool

Hermes should be used when the system needs:

```text
long-running task memory
operator preferences
project state recall
agent behavior experiments
multi-step synthesis
```

Hermes should not be the default for tiny exact-output tool calls.

Required bridge:

```text
OpenWebUI / MCPO
-> hermes_memory_query
-> hermes_agent_task
-> structured result back to chat and audit ledger
```

### 3. Ruflo Graduates From Planning-Only To Policy-Governed Orchestration

Current safe mode:

```text
ruflo_status
ruflo_plan
ruflo_execution_packet
```

Next mode:

```text
ruflo_orchestration_dry_run
ruflo_team_plan
ruflo_task_graph
ruflo_worker_packet
```

Later, after isolation proof:

```text
ruflo_execute_in_sandbox
```

Ruflo should not get raw production credentials. It should produce task graphs and worker packets that the capability broker executes only if policy allows.

### 4. Capability Broker Becomes The Enforcement Layer

The broker is the wall between agent intelligence and production power.

It should own:

```text
schemas
risk classification
rate limits
credential boundaries
audit logging
rollback metadata
approval rules
```

Models can request capabilities. They cannot improvise production actions.

### 5. Paperclip Becomes The Durable Work Ledger

Paperclip must support reliable scoped writes:

```text
create issue
post comment
set disposition
attach evidence
link commit / trace IDs
```

Until this is reliable, the system feels semi-manual. This is the biggest product unlock.

### 6. VPS Becomes An Execution Runtime, Not A Manual Console

The VPS should expose controlled capabilities:

```text
run_stack_health
run_rollback_drill
render_compose_config_redacted
deploy_canary
rollback_canary
collect_logs_redacted
```

No raw shell in chat. No raw Docker in model hands.

## Product Modes

### Mode 1: Operator Copilot

Human asks. Veloce plans, checks, and drafts.

Status: ready.

### Mode 2: Governed Operator Execution

Human approves. Veloce runs safe capability calls and records proof.

Status: mostly ready.

### Mode 3: Unattended Dry Run

Veloce runs the whole loop without changing production.

Status: ready in v1.8 dry-run.

### Mode 4: Unattended Comment-Only Production

Veloce can close low-risk Paperclip loops by posting evidence and disposition comments.

Status: next target.

### Mode 5: Unattended PR-Only Production

Veloce can create branches and PRs, but cannot merge.

Status: next target after Paperclip scoped auth.

### Mode 6: Unattended Canary Production

Veloce can deploy one approved low-risk canary, verify, and rollback.

Status: later target.

```text

No raw shell exposed.
No secrets printed.
```

### V1.9B - Hermes Memory Tool Bridge

Deliver `hermes_memory_query` and `hermes_agent_task` via MCPO/OpenWebUI.

Acceptance:

```text
Hermes can recall project/operator state.
Hermes output is structured.
Result is logged in audit ledger.
```

### V1.9C - Ruflo Orchestration Dry Run

Deliver Ruflo team/task graph generation without production execution.

Acceptance:

```text
Ruflo returns task graph.

Forbidden steps are denied with reasons.
```

```text
Post comment.

No recovery loop.
```

```text
run_stack_health

collect_logs_redacted
```

Deliver the investor demo:

```text

```

### V1.9I - Obsidian + Graphify Knowledge Layer

Turn Obsidian from a human-readable mirror into a graph-backed knowledge layer.

Acceptance:

```text
.graphifyignore excludes secrets and runtime state.
Graphify generates GRAPH_REPORT.md and graph.json.
Graph export is mirrored into Obsidian.
Future OpenWebUI endpoint can answer knowledge_graph_query from graph.json.
Hermes/Ruflo can use graph results as structured context without reading the whole repo.
```

### V1.9J - Knowledge Graph Memory Loop

Expose the graph as a product capability instead of a terminal artifact.

Acceptance:

```text
MCPO exposes knowledge_graph_status.
MCPO exposes knowledge_graph_query.
MCPO exposes knowledge_memory_record.
OpenWebUI imports all three graph-memory tools.
Memory record blocks secret-like content.
Memory record emits Obsidian-ready markdown for Graphify ingestion.
```

### V1.9K - Graph Query Quality Upgrade

Make graph answers useful to operators, not just developers reading raw node matches.

Acceptance:

```text
knowledge_graph_query supports source_filter, answer_style, and include_relationships.
Docs-first ranking returns architecture/runbook/product nodes before tests for broad questions.
source_filter=tests still returns test evidence when explicitly requested.
Responses preserve legacy raw graph fields and add summary_answer plus ranking metadata.
OpenWebUI prompt can request source_filter=docs for graph-grounded architecture answers.
```

### V1.9L - Operating Graph Ingestion Layer

Make the graph represent the operating system, not only the repo.

Acceptance:

```text
Graph-memory markdown files describe Paperclip, Hermes, Ruflo, OpenWebUI, MCPO, Obsidian, Graphify, GitHub, and VPS relationships.
OpenWebUI and MCPO tool surfaces are exported as graphable memory nodes.
Paperclip issue/task/team/disposition metadata can be imported from secret-free JSONL.
Obsidian can display the same memory files that Graphify extracts into graph.json.
Raw comments, secrets, tokens, and environment values remain excluded from graph ingestion.
```

### V1.9M - Operating Graph Ranking Boost

Make generated operating memory appear as top evidence for product graph questions.

Acceptance:

```text
knowledge/graph-memory is treated as a first-class graph source kind.
source_filter=docs includes both docs and generated graph-memory markdown.
source_filter=knowledge returns only generated graph-memory markdown.
Operating graph queries boost V1.9L and knowledge/graph-memory nodes above older setup/runbook docs.
OpenAPI documents knowledge as an accepted source_filter value.
```
