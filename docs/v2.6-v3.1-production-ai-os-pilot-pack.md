# V2.6-V3.1 Production AI OS Pilot Pack

Date: 2026-05-27

## Context

V2.1-V2.5 added the durable execution runner layer: typed MCPO execution endpoints, file-backed job packets, leases, heartbeats, audit tails, graph-memory events, and redaction checks.

V2.6-V3.1 turns the next production AI OS steps into one safe pilot pack. The pack does not enable every live action at once. It evaluates each capability independently, records what is ready, blocks what is not explicitly enabled, and writes evidence for OpenWebUI, Obsidian, Graphify, Paperclip, GitHub, and the VPS workflow.

## Run Command

```bash
cd projects/reliability-policy-matrix
```

Each stage also requires `live_enabled=true` in a local config copy before live action can proceed. The committed config remains dry-run safe.

## Safety Rules

- No endpoint or runner accepts raw shell, raw Docker, raw token, arbitrary filesystem paths, or untyped command text.
- Missing environment variables are reported by name only.
- Secret-like strings are redacted from markdown, audit output, graph memory, PR bodies, and comments.
- Paperclip and GitHub are first-class live pilots because they are lower risk than canary deploy or rollback.
- Canary and rollback remain protected gates until live credentials, health checks, alerting, and operator approvals are present.
- Approval changes candidate state; it does not directly execute a destructive production action.

## Product Meaning

The end-to-end shape is:

```text
OpenWebUI operator request
-> Veloce Agentic Control / MCPO
-> typed production AI OS pilot pack
-> capability-specific dry-run or gated live adapter
-> audit ledger and memory markdown
-> Graphify rebuild
-> OpenWebUI knowledge graph query
```

This is the bridge from "we can prove the architecture" to "we can safely run selected production actions one at a time."

## Current Status

V2.6-V3.1 is an operator-supervised production AI OS pilot pack. It is production-shaped and auditable, but the committed repository intentionally does not enable live mutation by default. Live capability rollout should happen one stage at a time with scoped credentials, local live config, a current graph-memory rebuild, and a rollback note attached to the evidence.
