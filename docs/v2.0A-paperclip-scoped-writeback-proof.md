# V2.0A Paperclip Scoped Writeback Proof

V2.0A is the first focused proof for live Paperclip writeback under the V2.0 production execution control plane.

The goal is intentionally narrow:

```text
one Paperclip issue -> one scoped comment -> one scoped disposition update -> audit ledger -> graph memory packet
```

This is not broad Paperclip administration and it is not unattended production autonomy. It is a controlled writeback proof that can be run dry, reviewed, and then promoted with explicit live gates.

## Default Dry Run

```bash
cd projects/reliability-policy-matrix
make paperclip-writeback-proof
```

The default config is:

```text
configs/paperclip_writeback_v2_0A.json
```

By default, `live_enabled` is `false`, so the script does not send network write requests.

## Live Gates

A live write is blocked unless all gates pass:

```text
VELOCE_PAPERCLIP_WRITEBACK_LIVE=1
live_enabled=true
PAPERCLIP_BASE_URL is set
PAPERCLIP_AUTOMATION_TOKEN is set
```

The token is used only as a bearer token and is not printed in generated outputs.

## Scoped Actions

The proof prepares:

```text
POST /api/issues/{issue_id}/comments
PATCH /api/issues/{issue_id}
```

These endpoint templates are configuration, not hard-coded policy. If the deployed Paperclip API uses different routes, update a local config copy and keep the committed default safe.

## Outputs

```text
artifacts/derived/paperclip_writeback_v2_0A.json
artifacts/derived/paperclip_writeback_v2_0A.md
artifacts/derived/paperclip_writeback_audit_v2_0A.jsonl
artifacts/derived/paperclip_writeback_memory_v2_0A.md
```

The memory markdown is intended for the Obsidian -> Graphify loop, so the result becomes queryable through OpenWebUI graph memory after extraction.

## Rollback

If a live proof writes the wrong comment or disposition, the rollback is:

```text
post a correction comment
restore the prior disposition
rerun graph extraction so memory reflects the correction
```

The V2.0A control boundary is small on purpose. Once this proof is accepted, the next layers are chat-to-PR and chat-to-canary under the same gated pattern.
